**Test 1: Escalation progression (SILENT → AUDIBLE → FULLSCREEN)**

```
Scenario: Reminder fires → user ignores → escalation progresses through 3 tiers
```

Steps:

1. Seed a reminder scheduled to fire in 30 seconds
2. Wait for notification to appear (silent tier — notification only, no sound)
3. Do NOT interact — wait for escalation timeout (2 min default)
4. Verify notification upgrades to audible tier (sound + vibration indicators)
5. Wait for next escalation timeout (3 min default)
6. Verify full-screen intent fires (or high-priority notification if permission denied)
7. Confirm the reminder to stop escalation

Assertions:

- First notification appears within reasonable time of schedule
- Escalation progresses if user doesn't act
- Full-screen alarm screen appears (or heads-up notification as fallback)
- Confirming at any tier stops the escalation

**NOTE:** This test takes ~6 minutes due to escalation waits. For CI, consider reducing escalation timeouts via test override (e.g., 10s silent → 10s audible → fullscreen).

**Test 2: Notification action buttons work from tray**

```
Scenario: Notification shows DONE/SNOOZE/SKIP buttons — each works without opening app
```

Steps:

1. Seed a reminder, wait for notification
2. Open notification shade via `$.native.openNotifications()`
3. Verify notification has DONE, SNOOZE, SKIP action buttons
4. Tap DONE
5. Verify notification is dismissed
6. Open app → verify reminder status is DONE
7. Repeat for SNOOZE and SKIP in separate sub-tests

Assertions:

- All 3 action buttons are present on the notification
- Tapping each button updates the reminder state in the database
- SNOOZE reschedules the notification for later
- App state reflects the action taken

**Test 3: Notifications persist until user acts**

```
Scenario: Notification fires → user reads but doesn't tap action → notification stays
```

Steps:

1. Seed a reminder, wait for notification
2. Open and close notification shade multiple times
3. Wait 5 minutes
4. Open notification shade again
5. Verify notification is still present (not auto-dismissed)

Assertions:

- Notification persists after being shown
- Notification is NOT marked as auto-cancel
- Notification only disappears when user taps an action button

```

```

1. Start onboarding

5. Verify alarm functionality is degraded but app doesn't crash

Assertions:

- App handles denial gracefully (no crash)
- User sees clear messaging about what won't work
- Core functionality (data entry, chain management) still works
- Permission can be re-requested later

4. Return to app
5. Verify app detects the newly granted permission

Assertions:

- App detects permission change on resume
- Previously degraded features now work
- No app restart required
