<objective>
Wire AnchorResolver into the Riverpod state management layer via AnchorNotifier. This is the integration plan that connects user actions (confirm meal, change default time) to domain logic (AnchorResolver) and platform services (AlarmScheduler + Drift).

AnchorNotifier is the single orchestrator for all anchor-related operations:

1. Confirm a meal → updates anchor → resolves dependents → reschedules alarms
2. Change default time → updates anchor → cascades recalculation to all dependents

This completes the Phase 05 data flow:

```
User confirms "I had lunch" at 1:15 PM
  → AnchorNotifier.confirmMeal('lunch', 1:15 PM UTC)
    → AnchorRepository.updateAnchor(confirmed_at: 1:15 PM)
    → ReminderRepository.watchForAnchorDependents('lunch')
    → AnchorResolver.resolve(anchor, 1:15 PM, dependents)
    → For each ReminderScheduleUpdate:
      → ReminderRepository.updateReminder(scheduledAt: computed)
      → AlarmScheduler.schedule(reminder)
```

**Key constraint from ARCHITECTURE.md §7 (Anti-Patterns):**

> The chain engine never directly calls platform services. The application layer
> (Riverpod notifiers) orchestrates domain → platform handoff.

AnchorNotifier is the application layer that orchestrates AnchorResolver (domain) → AlarmScheduler (platform).

**Dependent reminder query strategy:**
ReminderRepository needs a method to find reminders that are meal-anchored for a specific meal type. Options:

1. Add `watchRemindersForMealType(String mealType)` to ReminderRepository (clean but requires DAO change)
2. Use existing `watchForChain()` and filter by medicineType client-side (works but less efficient)
3. Query all active reminders and filter by meal-dependent types (simplest, acceptable at scale for <50 reminders per user)

**Decision: Option 3** — query all active reminders via `watchActive()` and filter where `medicineType` is beforeMeal, afterMeal, or emptyStomach. For doseGap, the trigger is a dose confirmation (handled by ConfirmationNotifier in Phase 04), not a meal confirmation. However, the resolver still handles doseGap for completeness if a doseGap reminder is passed as a dependent.

Actually, looking at the data model more carefully: reminders don't have a direct `anchorMealType` field — they depend on anchors implicitly through the chain DAG structure. The chain engine determines which reminders are activated when an anchor is confirmed.

**Revised approach:** AnchorNotifier.confirmMeal() should:

1. Update anchor.confirmed_at
2. Query ALL active reminders (the list is small — <50 per user)
3. Filter to meal-dependent types (beforeMeal, afterMeal, emptyStomach, doseGap)
4. Pass them to AnchorResolver.resolve()
5. Batch update + reschedule

This is simple, correct, and avoids over-engineering a meal-type → reminder mapping that the schema doesn't natively support.

```dart
import 'package:riverpod_annotation/riverpod_annotation.dart';

import '../../../core/platform/alarm_scheduler.dart';
import '../data/anchor_repository.dart';
import '../domain/anchor_resolver.dart';
import '../domain/models/meal_anchor.dart';
import '../../reminders/data/reminder_repository.dart';
import '../../reminders/domain/models/medicine_type.dart';

import 'providers.dart';

part 'anchor_notifier.g.dart';

/// Manages meal anchor state — confirming meals and updating default times.
///
/// This is the integration orchestrator for the anchor resolution pipeline:
/// confirm meal → resolve dependent times → update Drift → reschedule alarms.
///
/// Satisfies ANCR-01 (configurable times), ANCR-02 (confirm → recalculate).
@riverpod
class AnchorNotifier extends _$AnchorNotifier {
  @override
  Future<List<MealAnchor>> build() async {
    // Watch the reactive stream of all meal anchors
    final anchors = ref.watch(allAnchorsProvider);
    return anchors.when(
      data: (data) => data,
      loading: () => [],
      error: (e, st) => throw e,
    );
  }

  /// Confirm that the user had a meal at [confirmedAt].
  ///
  /// Flow:
  /// 1. Update anchor.confirmed_at in Drift
  /// 2. Query all active reminders with meal-dependent medicine types
  /// 3. AnchorResolver.resolve() → compute new fire times
  /// 4. Batch update reminder.scheduled_at in Drift
  /// 5. AlarmScheduler.schedule() for each affected reminder
  ///
  /// [mealType] is 'breakfast', 'lunch', or 'dinner'.
  /// [confirmedAt] is the UTC timestamp when the meal was confirmed.
  Future<void> confirmMeal({
    required String mealType,
    required DateTime confirmedAt,
  }) async {
    state = const AsyncLoading();
    state = await AsyncValue.guard(() async {
      final anchorRepo = ref.read(anchorRepositoryProvider);
      final reminderRepo = ref.read(reminderRepositoryProvider);
      final resolver = ref.read(anchorResolverProvider);
      final alarmScheduler = ref.read(alarmSchedulerProvider);

      // 1. Get current anchor and update confirmed_at
      final anchors = await anchorRepo.watchAll().first;
      final anchor = anchors.firstWhere(
        (a) => a.mealType == mealType,
        orElse: () => throw ArgumentError('Unknown meal type: $mealType'),
      );
      final updatedAnchor = anchor.copyWith(confirmedAt: confirmedAt);
      await anchorRepo.updateAnchor(updatedAnchor);

      // 2. Query all active reminders that depend on meal anchors
      final allReminders = await reminderRepo.watchActive().first;
      final mealDependents = allReminders.where(_isMealDependent).toList();

      if (mealDependents.isNotEmpty) {
        // 3. Resolve new fire times
        final updates = resolver.resolve(
          anchor: updatedAnchor,
          confirmedAt: confirmedAt,
          dependents: mealDependents,
        );

        // 4 & 5. Batch update Drift + reschedule alarms
        for (final update in updates) {
          final reminder = mealDependents.firstWhere(
            (r) => r.id == update.reminderId,
          );
          final updatedReminder = reminder.copyWith(
            scheduledAt: update.scheduledAt,
          );
          await reminderRepo.updateReminder(updatedReminder);
          await alarmScheduler.schedule(updatedReminder);
        }
      }

      // Return updated anchors for state
      return await anchorRepo.watchAll().first;
    });
  }

  /// Update the default time for a meal anchor (ANCR-01).
  ///
  /// [minutesFromMidnight] is the new default time (e.g., 480 = 08:00 AM).
  /// After updating, cascades recalculation to all meal-dependent reminders
  /// using the new default time as the anchor reference (success criterion 4).
  Future<void> updateDefaultTime({
    required String mealType,
    required int minutesFromMidnight,
  }) async {
    state = const AsyncLoading();
    state = await AsyncValue.guard(() async {
      final anchorRepo = ref.read(anchorRepositoryProvider);
      final reminderRepo = ref.read(reminderRepositoryProvider);
      final resolver = ref.read(anchorResolverProvider);
      final alarmScheduler = ref.read(alarmSchedulerProvider);

      // 1. Get current anchor and update default time
      final anchors = await anchorRepo.watchAll().first;
      final anchor = anchors.firstWhere(
        (a) => a.mealType == mealType,
        orElse: () => throw ArgumentError('Unknown meal type: $mealType'),
      );
      final updatedAnchor = anchor.copyWith(
        defaultTimeMinutes: minutesFromMidnight,
      );
      await anchorRepo.updateAnchor(updatedAnchor);

      // 2. Cascade: recalculate dependent reminders using the new default time
      // Convert minutes-from-midnight to today's DateTime for resolution
      final now = DateTime.now().toUtc();
      final referenceTime = DateTime.utc(
        now.year,
        now.month,
        now.day,
        minutesFromMidnight ~/ 60,
        minutesFromMidnight % 60,
      );

      final allReminders = await reminderRepo.watchActive().first;
      final mealDependents = allReminders.where(_isMealDependent).toList();

      if (mealDependents.isNotEmpty) {
        final updates = resolver.resolve(
          anchor: updatedAnchor,
          confirmedAt: referenceTime,
          dependents: mealDependents,
        );

        for (final update in updates) {
          final reminder = mealDependents.firstWhere(
            (r) => r.id == update.reminderId,
          );
          final updatedReminder = reminder.copyWith(
            scheduledAt: update.scheduledAt,
          );
          await reminderRepo.updateReminder(updatedReminder);
          await alarmScheduler.schedule(updatedReminder);
        }
      }

      return await anchorRepo.watchAll().first;
    });
  }

  /// Whether a reminder depends on meal anchors for its scheduled time.
  static bool _isMealDependent(Reminder r) => switch (r.medicineType) {
        MedicineType.beforeMeal => true,
        MedicineType.afterMeal => true,
        MedicineType.emptyStomach => true,
        MedicineType.doseGap => true,
        MedicineType.fixedTime => false,
      };
}
```

**Key design decisions:**

- NOT keyed per anchor (all 3 anchors in one notifier) — simpler state management for 3 fixed entities.
- `confirmMeal` and `updateDefaultTime` both follow the same cascade pattern: update anchor → resolve → update reminders → reschedule.
- Uses `AsyncValue.guard()` for error handling — wraps the entire flow, sets AsyncError on failure.
- `_isMealDependent` filter ensures fixedTime reminders are never passed to resolver (belt and suspenders — resolver also excludes them).
- `watchActive().first` gets a snapshot — not a persistent subscription. This is intentional: we want point-in-time data, not a reactive stream inside the mutation.
- `alarmScheduler.schedule()` handles both "schedule new" and "cancel old + schedule new" — the scheduler is idempotent per reminder ID.
- Provider reads use `ref.read()` (not `ref.watch()`) inside mutation methods — watch is for build(), read is for imperative actions.
